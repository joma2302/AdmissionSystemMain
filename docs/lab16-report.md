# ЗВІТ З НАВЧАЛЬНОЇ ПРАКТИКИ
## Лабораторна робота №16
### Тема: Інтеграція бізнес-логіки до адміністративної частини додатку та тестування

---

### 1. Вступ
**Напрям дослідження:** Інтеграція складної бізнес-логіки в користувацький інтерфейс адміністратора, забезпечення цілісності процесів зарахування та розробка стратегії автоматизованого тестування (модульного та інтеграційного).

**Мета роботи:** Реалізувати повний цикл інтеграції логіки зарахування абітурієнтів до адміністративної панелі, враховуючи предметні вимоги факультетів, та забезпечити високе покриття коду тестами для гарантування коректності алгоритмів ранжування.

**Завдання:**
1. Розширити доменну модель `AdmissionSheet` для підтримки перевірки мінімальних балів з профільних предметів.
2. Інтегрувати оновлену логіку в `AdmissionService` та `AdminController`.
3. Розробити набір модульних тестів для `AdmissionSheet` та `AdmissionService`.
4. Створити інтеграційні тести для перевірки веб-шару (`AdminController`) з використанням MockMvc.

### 2. Лаконічний опис результатів виконання
В ході лабораторної роботи було завершено інтеграцію бізнес-логіки "Приймальної комісії" з адміністративним інтерфейсом.

**Ключові результати:**
- **Доменна логіка:** Удосконалено алгоритм зарахування. Тепер система не просто обирає абітурієнтів з найвищим балом, а й перевіряє, чи відповідають їхні оцінки мінімальним вимогам факультету (наприклад, не менше 10 балів з математики).
- **Сервісний шар:** `AdmissionService` тепер коректно оркеструє процес зарахування, оновлюючи статуси заявок у базі даних після проведення розрахунків через `AdmissionSheet`.
- **Тестування:** 
    - Створено `AdmissionSheetTest` для перевірки алгоритму ранжування (4/4 тести пройдено).
    - Створено `AdmissionServiceTest` з використанням Mockito для перевірки бізнес-сценаріїв у сервісі (3/3 тести пройдено).
    - Впроваджено `AdminControllerTest` для тестування маршрутів адмін-панелі та інтеграції з безпекою.

### 3. Текст програми з коментарями

#### 3.1. Оновлена логіка зарахування (AdmissionSheet.java)
```java
public void determineAdmitted() {
    // Сортування заявок за спаданням сумарного балу
    List<Application> sorted = applications.stream()
            .sorted(Comparator.comparingInt(Application::getTotalScore).reversed())
            .collect(Collectors.toList());

    int admittedCount = 0;
    for (Application app : sorted) {
        // Зарахування лише при наявності місць ТА відповідності вимогам
        if (admittedCount < faculty.getMaxStudents() && meetsRequirements(app)) {
            app.admit();
            admittedCount++;
        } else {
            app.reject();
        }
    }
}

private boolean meetsRequirements(Application app) {
    List<SubjectRequirement> reqs = faculty.getRequirements();
    if (reqs.isEmpty()) return true; // Якщо вимог немає — проходить будь-який бал

    List<Grade> grades = app.getApplicant().getGrades();
    for (SubjectRequirement req : reqs) {
        // Перевірка наявності оцінки з потрібного предмету з балом >= мінімального
        boolean satisfied = grades.stream()
                .anyMatch(g -> g.getSubject() == req.getSubject() && g.getScore() >= req.getMinimumScore());
        if (!satisfied) return false;
    }
    return true;
}
```

### 5. Змістовний аналіз та висновки

**Аналіз результатів:**
Інтеграція бізнес-логіки дозволила автоматизувати найбільш критичний процес — прийняття рішень про зарахування. Використання паттерну **Strategy** (через `AdmissionSheet`) відокремило алгоритм зарахування від інфраструктурного коду. Впровадження перевірки вимог до предметів зробило систему більш гнучкою та відповідною реальним правилам вступу. Тестування за допомогою Mockito дозволило ізолювати бізнес-логіку від бази даних, що пришвидшило розробку.

**Висновки:**
Лабораторна робота №16 підтвердила ефективність архітектурного підходу, де бізнес-правила інкапсульовані в доменних моделях. Система успішно обробляє складні сценарії зарахування, а автоматизовані тести гарантують стабільність коду при подальших змінах. Адміністративна частина додатку стала функціонально завершеною та готовою до експлуатації.
