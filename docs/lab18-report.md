# ЗВІТ З НАВЧАЛЬНОЇ ПРАКТИКИ
## Лабораторна робота №18
### Тема: Інтеграція клієнтської та адміністративної частини у загальний проект

---

### 1. Вступ
**Напрям дослідження:** Веб-розробка на базі Spring Boot, інтеграція багатокористувацьких інтерфейсів, забезпечення безпеки та розмежування прав доступу в інформаційних системах.

**Мета роботи:** Об'єднати розроблені клієнтську та адміністративну частини інформаційної системи "Приймальна комісія" в єдиний цілісний проект, забезпечити коректну взаємодію між ними та перевірити відсутність конфліктів.

**Завдання:**
1. Перенести всі контролери, сервіси та шаблони в єдину структуру проекту.
2. Налаштувати спільну систему навігації через базовий макет (layout).
3. Перевірити коректність роботи Spring Security при переході між клієнтськими та адміністративними маршрутами.
4. Провести повне інтеграційне тестування об'єднаної системи.

---

### 2. Лаконічний опис результатів виконання
В ході лабораторної роботи було завершено фінальну інтеграцію системи.

**Ключові результати:**
- **Архітектурна єдність:** Всі компоненти системи тепер працюють у межах одного Spring Boot додатку. Клієнтська частина (реєстрація, подача заявок) та адміністративна частина (керування факультетами, статусами та зарахування) використовують спільний доменний шар та шар персистентності.
- **Динамічна навігація:** У файлі `layout.ftl` реалізовано логіку відображення меню на основі ролі авторизованого користувача. Адміністратори бачать посилання на панель керування, а абітурієнти — на свої заявки та профіль.
- **Безпека:** Конфігурація `SecurityConfig` забезпечує захист маршрутів `/admin/**` та `/admission/**`, дозволяючи доступ до них лише користувачам з роллю `ROLE_ADMIN`.
- **Тестування:** Виконано повний набір тестів (53 тести), що підтверджує стабільність об'єднаної системи та відсутність регресійних помилок.

### 3. Текст програми з коментарями

#### 3.1. Спільна навігація (layout.ftl)
```html
<div class="navbar">
    <div class="nav-links">
        <a href="/" class="nav-brand">Вступ 2026</a>
        <a href="/">Головна</a>

        <#-- Динамічне меню залежно від авторизації та ролі -->
        <#if isAuthenticated>
            <#if isAdmin>
                <#-- Адміністративні посилання -->
                <a href="/admin">Адмін-панель</a>
                <a href="/admin/applications">Заявки</a>
                <a href="/admin/faculties">Факультети</a>
                <a href="/admission">Зарахування</a>
            <#else>
                <#-- Клієнтські посилання -->
                <a href="/register">Мій профіль</a>
                <a href="/apply">Подати заявку</a>
                <a href="/applications">Мої заявки</a>
            </#if>
            <a href="/ranking">Загальний рейтинг</a>
        </#if>
    </div>
    <#-- Блок інформації про користувача та вихід -->
</div>
```

#### 3.2. Розмежування прав доступу (SecurityConfig.java)
```java
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http.authorizeHttpRequests(auth -> auth
        // Публічні ресурси
        .requestMatchers("/", "/signup", "/login", "/css/**").permitAll()
        // Захищені адміністративні маршрути
        .requestMatchers("/admin/**", "/admission/**").hasRole("ADMIN")
        // Решта функцій потребує входу в систему
        .anyRequest().authenticated()
    );
    // Налаштування форми входу та логауту...
    return http.build();
}
```

### 4. Вхідні та вихідні дані програми

**Вхідні дані:**
- Облікові дані користувачів з різними ролями (ADMIN, USER).
- Заявки абітурієнтів, подані через клієнтський інтерфейс.
- Налаштування факультетів та їх лімітів.

**Вихідні дані:**
- Цілісний веб-інтерфейс, що адаптується під роль користувача.
- Змінені статуси заявок в БД після дій адміністратора.
- Результати успішного проходження 53-х модульних та інтеграційних тестів.

### 5. Змістовний аналіз та висновки

**Аналіз результатів:**
Інтеграція пройшла успішно завдяки використанню патерну "Layout" у Freemarker, що дозволило уніфікувати зовнішній вигляд системи, зберігаючи специфічний функціонал для різних ролей. Використання Spring Security як центрального механізму контролю доступу усунуло конфлікти між клієнтською та адмін-частинами. Дані, введені абітурієнтом, миттєво стають доступними для обробки адміністратором, що демонструє цілісність бізнес-процесів системи.

**Висновки:**
В ході виконання лабораторної роботи №18 було успішно об'єднано всі модулі інформаційної системи в єдиний проект. Система відповідає вимогам щодо безпеки, зручності навігації та функціональної повноти. Проведене тестування підтвердило готовність проекту до експлуатації як комплексного рішення для автоматизації роботи приймальної комісії.
