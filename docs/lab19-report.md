# ЗВІТ З НАВЧАЛЬНОЇ ПРАКТИКИ
## Лабораторна робота №19
### Тема: Забезпечення безпеки додатків

---

### 1. Вступ
**Напрям дослідження:** Інформаційна безпека веб-додатків, керування доступом на основі ролей (RBAC) у середовищі Spring Framework.

**Мета роботи:** Налаштувати систему автентифікації та авторизації користувачів, впровадити розмежування прав доступу для різних ролей (адміністратор, менеджер, абітурієнт) та забезпечити захист веб-ресурсів додатка.

**Завдання:**
1. Створити конфігураційний файл `MvcConfig` для налаштування маппінгів представлень.
2. Налаштувати `SecurityConfig` (WebSecurityConfig) для роботи зі Spring Security.
3. Впровадити ролі `ADMIN`, `MANAGER` та `APPLICANT`.
4. Забезпечити збереження облікових даних та ролей у базі даних.
5. Налаштувати захист маршрутів залежно від прав доступу.

---

### 2. Лаконічний опис результатів виконання
В ході лабораторної роботи було реалізовано повноцінну систему безпеки додатка.

**Ключові результати:**
- **Конфігурація MVC:** Створено `MvcConfig`, який спрощує маршрутизацію для стандартних сторінок (наприклад, сторінки входу).
- **Розширення моделі ролей:** До системи додано нову роль `MANAGER`, яка має права на керування заявками та перегляд адміністративної панелі, аналогічно до `ADMIN`.
- **Захист маршрутів:** Оновлено `SecurityConfig`, де налаштовано `SecurityFilterChain`. Маршрути `/admin/**` та `/admission/**` тепер доступні лише користувачам з ролями `ADMIN` або `MANAGER`.
- **Автоматичне створення користувачів:** В `AuthService` реалізовано `CommandLineRunner`, який при старті системи перевіряє наявність адміністратора та менеджера за замовчуванням і створює їх у разі відсутності.
- **Інтерфейс користувача:** Шаблон `layout.ftl` адаптовано для коректного відображення навігації та статусних бейджів для нової ролі `MANAGER`.

### 3. Текст програми з коментарями

#### 3.1. Конфігурація Spring Security (SecurityConfig.java)
```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                // Публічні ресурси доступні всім
                .requestMatchers("/", "/signup", "/login", "/css/**").permitAll()
                // Захищені адміністративні маршрути для ADMIN та MANAGER
                .requestMatchers("/admin/**", "/admission/**").hasAnyRole("ADMIN", "MANAGER")
                // Решта функцій потребує авторизації
                .anyRequest().authenticated()
            )
            .formLogin(form -> form
                .loginPage("/login")
                .defaultSuccessUrl("/", true)
                .permitAll()
            )
            // Налаштування Logout, CSRF та інше...
        return http.build();
    }
}
```

#### 3.2. Конфігурація MVC (MvcConfig.java)
```java
@Configuration
public class MvcConfig implements WebMvcConfigurer {
    @Override
    public void addViewControllers(ViewControllerRegistry registry) {
        // Прямий маппінг URL на назву шаблону без контролера
        registry.addViewController("/login").setViewName("login");
    }
}
```

### 4. Вхідні та вихідні дані програми

**Вхідні дані:**
- Логін та пароль користувача при вході.
- Запити до захищених URL-адрес.
- Дані про ролі в таблиці `users` бази даних.

**Вихідні дані:**
- Успішна авторизація та перенаправлення на головну сторінку.
- Повідомлення про помилку при неправильних даних або відсутності прав (403 Forbidden).
- Динамічне меню, що приховує/показує пункти залежно від ролі.

---

### 5. Змістовний аналіз та висновки

**Аналіз результатів:**
Використання Spring Security дозволило централізовано керувати безпекою додатка. Впровадження ролі `MANAGER` продемонструвало гнучкість налаштованої архітектури — додавання нових рівнів доступу вимагає мінімальних змін у коді конфігурації. Збереження ролей у БД забезпечує стабільність системи та можливість легкого масштабування штату користувачів з різними правами.

**Висновки:**
В ході виконання лабораторної роботи №19 було успішно забезпечено безпеку інформаційної системи. Додаток надійно захищений від несанкціонованого доступу, реалізовано зручний механізм автентифікації та гнучку систему розмежування прав доступу на основі ролей.
